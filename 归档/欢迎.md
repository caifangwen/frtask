	好问题。任务数量不确定的时候，"三桌面"这种固定结构就不够用了，需要一套**可扩展的调度框架**，而不是针对具体任务的布局。

---

## 核心思路转变

固定布局 → **动态分层**

不再给每个任务分配一个桌面，而是按**任务的状态**来分层：

```
不是问"这个任务在哪里"
而是问"这个任务现在处于什么状态"
```

---

## 通用框架：三层状态分类

任何任务在任何时刻，只会处于三种状态之一：

```
🔴 活跃层（Active）     ← 我现在正在做
🟡 等待层（Pending）    ← 在等某个结果（AI回复/加载/他人反馈）
⚪ 冻结层（Frozen）     ← 今天暂时不碰，但没完成
```

**桌面的划分对应状态，而不是对应任务：**

```
桌面 1 = 活跃层   当前专注的唯一任务
桌面 2 = 等待层   所有挂起的任务窗口堆在这里
桌面 3 = 资源层   文档、参考资料、工具（不干活，只查）
```

任务在三个桌面之间流动，而不是固定在某一个桌面。

---

## 调度流程

### 日常运转逻辑

```
开始一个任务
    ↓
放入 桌面1（活跃）全力执行
    ↓
遇到等待（AI/加载/阻塞）
    ↓
把这个任务的窗口拖到 桌面2（等待）
写一句话：[任务名] 回来后→ 做什么
    ↓
从桌面2 的其他任务里，挑一个认知负荷匹配的
拖到 桌面1，继续执行
    ↓
收到通知（AI回复/加载完成）
    ↓
从桌面2 捞回，拖到 桌面1
```

---

### 选择"下一个任务"的决策树

等待期不超过 2 分钟：
```
→ 去桌面3 查资料，或做当前任务的准备工作
→ 不要切换到新任务，切换成本大于收益
```

等待期 2-10 分钟：
```
→ 去等待层里挑一个「低认知任务」
→ 机械操作、参数对照、视觉检查类
→ 不要碰需要深度逻辑的任务
```

等待期超过 10 分钟：
```
→ 可以切换到任意任务
→ 但必须先写好断点锚点再切
```

---

## 断点锚点：统一格式

所有任务共用一个锚点文件（一个置顶的纯文本便签），格式固定：

```
[任务名] 状态：等待AI / 等待加载 / 我去吃饭了
         回来做：___________________
         注意：（可选，写坑或关键变量）
```

**例子：**
```
[翻译程序]  状态：AI处理中
            回来做：把正则结果贴进 test_format() 跑一遍
            注意：<span> 嵌套层级别超过3层会断

[WP导入]    状态：我去切任务了
            回来做：继续核对 post_meta 第12行
            
[客户邮件]  状态：等对方回复
            回来做：收到回复后看第二段要求再改方案
```

**这个文件是唯一需要维护的东西。** 其他什么看板、卡片、标签都是多余的。

---

## 工具选择（按优先级）

**必须有：**
- 虚拟桌面（系统自带，零成本）
- 一个置顶便签文件（Notepad++置顶 / macOS Stickies / Windows Sticky Notes）
- 剪贴板历史（Win+V 或 CopyQ）

**值得装：**
- **Raycast / PowerToys** → 快捷键直达任何窗口或文件，不用找
- **浏览器通知开启** → AI 回复完成自动弹窗，不用盯着看

**不需要：**
- 专门的项目管理软件（Notion/Jira 用来管任务可以，但不要放在调度流程里，太重）
- 多显示器不一定有帮助，虚拟桌面的切换比转头看屏幕认知负荷更低

---

## 可扩展性在哪里

任务从 3 个变成 10 个，这套框架不会崩溃，因为：

```
活跃层永远只有 1 个任务          ← 上限固定
等待层可以堆任意多个任务         ← 自动扩展
锚点文件只是多几行字             ← 线性增长
```

唯一需要控制的是：**等待层不能超过 5 个任务**。超过了说明你开了太多线，
这时候应该把最不紧急的任务整体冻结，关掉窗口，只在锚点文件里留一行记录，等有空再恢复。

---

## 一句话总结

> 按任务状态（活跃/等待/冻结）而不是按任务内容来划分桌面，活跃层永远只有一个任务，切换前写一句"回来做什么"，用等待期的长短决定切去做什么量级的事。
